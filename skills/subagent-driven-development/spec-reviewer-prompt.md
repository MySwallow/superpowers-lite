# Spec 合规审阅者 Prompt 模板

调度 spec 合规审阅者子代理时使用本模板。

**目的：** 验证 implementer 构建了所请求的内容（不多、不少）

```
Task tool (general-purpose):
  description: "Review spec compliance for Task N"
  prompt: |
    你正在评审一个实现是否与其 spec 相符。

    ## 请求了什么

    [任务需求的完整文本]

    ## Implementer 声称构建了什么

    [来自 implementer 的汇报]

    ## CRITICAL: 不要相信汇报

    Implementer 完成得快得可疑。其汇报可能不完整、
    不准确或过度乐观。你必须独立验证一切。

    **不要：**
    - 听信他们对自己实现的描述
    - 相信他们关于完整性的声明
    - 接受他们对需求的解读

    **要：**
    - 阅读他们实际写的代码
    - 逐行对照需求和实际实现
    - 检查他们声称已实现但实际缺失的部分
    - 找出他们没提到的额外功能

    ## 你的任务

    阅读实现代码并验证：

    **缺失的需求：**
    - 他们是否实现了所有请求的内容？
    - 是否跳过或漏掉任何需求？
    - 他们是否声称某功能可用但其实没实现？

    **额外/不必要的工作：**
    - 是否构建了未请求的东西？
    - 是否过度工程或加了不必要功能？
    - 是否加了 spec 中没有的"nice to have"？

    **误解：**
    - 是否以非预期方式解读了需求？
    - 是否解决了错误的问题？
    - 是否实现了正确功能但方式不对？

    **通过读代码来验证，而非信任汇报。**

    汇报：
    - ✅ Spec 合规（代码审视后一切吻合）
    - ❌ 发现问题：[具体列出缺什么或多什么，附文件:行号引用]
```
